    MkDocs + GitHub Pages 知识库部署使用手册¶

    文档类型: 部署指南 | 更新时间: 2026-06-04 | 维护人: 研发团队 | 标签: MkDocs, GitHub Pages, 知识库部署, Material, Git

    适用场景：团队内部知识库搭建，GitHub 开源仓库 Pages 永久免费托管，可选 Gitee 双仓库源码备份

    逻辑拆分：【初次搭建部署全流程】+【日常新增文档上线流程】+【常用命令速查】

    一、初次从零搭建知识库全流程¶

    说明

    本节内容仅在首次部署时执行一次，后续日常使用请参考第二章。

    环境前置安装¶

    安装清单

    • Git：官网下载
    • Python 3.10+：官网下载

    安装完成后打开 Git Bash，执行以下命令安装依赖：

    # 安装 mkdocs + material 美化主题
pip install mkdocs mkdocs-material

# 验证安装成功
mkdocs --version

    平台新建仓库¶

    平台 操作说明
    GitHub 新建仓库，仓库名自定义（示例 mkdocs），权限选择 Public 公开，不勾选 初始化 README
    Gitee（可选） 同步创建同名仓库，用于国内源码异地备份

    本地初始化 MkDocs 项目¶

    # 生成项目目录
mkdocs new mkdocs

# 进入项目根目录
cd mkdocs

    项目目录结构¶

    mkdocs/
├── mkdocs.yml      # 网站配置文件：站点名称、主题、侧边导航
└── docs/           # 所有 Markdown 文档存放目录，index.md = 网站首页
    推荐：添加 .gitignore 文件

    在项目根目录创建 .gitignore，避免提交不必要的文件：

    # MkDocs 生成文件
site/

# Python 缓存
__pycache__/
*.pyc
*.pyo
*.pyd

# 编辑器配置
.vscode/
.idea/
*.swp
*.swo

# 系统文件
.DS_Store
Thumbs.db

    基础配置¶

    编辑 mkdocs.yml，参考以下基础模板配置：

    site_name: 部门内部知识库

theme:
  name: material
  language: zh

nav:
  - 首页: index.md

    绑定远程 Git 仓库¶

    # 清空原有远程配置（如有）
git remote remove origin

# 绑定主仓库 GitHub（请替换为你的仓库地址）
git remote add origin https://github.com/<你的GitHub用户名>/mkdocs.git

# 【可选】绑定 Gitee 备用仓库
git remote add gitee https://gitee.com/<你的Gitee用户名>/mkdocs.git

# 查看绑定结果
git remote -v

    首次提交源码至 main 分支¶

    git add .
git commit -m "初始化知识库项目框架"

# 推送源码到 GitHub
git push origin main

# 【可选】同步源码到 Gitee
git push gitee main

    一键编译部署网页¶

    # --clean 参数：清理历史缓存，规范上线必带
mkdocs gh-deploy --clean

    Warning

    命令执行后自动打包静态网页、创建/更新 gh-pages 分支并自动推送 GitHub。

    GitHub 后台开启 Pages¶

    1. 进入仓库主页 → Settings → Pages
    2. Build and deployment 配置：
    3. Branch：gh-pages
    4. Folder：/(root)
    5. 点击 Save 保存，等待 5~15 分钟部署生效

    访问地址

    在线访问：https://<你的GitHub用户名>.github.io/mkdocs/

    二、日常新增/修改文档上线流程¶

    标准四步：写文档 → 本地预览校验 → 源码归档提交 → 外网正式发布

    步骤 1：编写文档¶

    1. 在 docs/ 目录新建/修改 .md 格式 Markdown 文档
    2. 打开 mkdocs.yml，在 nav 导航配置中追加文档目录：
    nav:
  - 首页: index.md
  - Linux部署文档: linux_deploy.md

    步骤 2：本地预览 ¶

    mkdocs serve

    浏览器访问 http://127.0.0.1:8000，确认页面、导航正常后关闭预览终端。

    步骤 3：Git 提交源码归档¶

    # 捕获全部修改文件
git add .

# 填写更新备注
git commit -m "新增：Linux服务部署操作文档"

# 推送源码至 GitHub 主分支 main
git push origin main

# 【可选同步备份】推送 Gitee
git push gitee main

    步骤 4：编译发布至外网站点¶

    mkdocs gh-deploy --clean

    执行完成后等待 3~10 分钟，GitHub Pages 站点自动刷新内容。

    三、常用辅助命令速查表¶

    功能说明 命令
    异地拉取最新项目源码 git pull origin main
    仅本地打包生成 site 静态文件夹 mkdocs build
    单独推送源码到 Gitee 仓库 git push gitee main

    四、关键使用说明¶

    • 费用规则

      GitHub 公开仓库 Pages 永久免费，站点存储上限 1GB，每月免费流量 100GB，内部知识库完全够用。

      注意

      私有仓库无法免费开通 Pages。

    • 访问 404 问题

      刚完成部署出现 404 为 GitHub 后台部署缓存，等待片刻刷新即可。

    • 临时极简用法

      仅临时修改内容、不需要归档源码，可跳过 git 提交：

      mkdocs serve
mkdocs gh-deploy --clean

    可选：自定义域名绑定（CNAME）¶

    如需使用自定义域名（如 docs.yourcompany.com），按以下步骤操作：

    在项目 docs/ 目录创建 CNAME 文件（无后缀名），写入你的域名： 

    docs.yourcompany.com

    在域名服务商添加 CNAME 解析：

    记录类型 主机记录 记录值
    CNAME docs（或你需要的子域名） <你的用户名>.github.io

    在 Settings → Pages → Custom domain 中填入你的域名，保存后会自动生成 CNAME 文件并提交到 gh-pages 分支。

    注意事项

    自定义域名配置后，CNAME 文件需要同时存在于 main 分支的 docs/ 目录和 gh-pages 分支的根目录。

    五、隐藏文件夹/文档配置说明¶

    Info

    如果某些文件夹或文档不希望在外网站点中显示（如内部设计文档、敏感资料等），可通过 mkdocs.yml 的 exclude_docs 配置实现隐藏。

    配置方法¶

    在 mkdocs.yml 中添加 exclude_docs 配置项，支持通配符匹配：

    plugins:
  - search
exclude_docs: |
  04_研发内部设计/**
  05_附件资源/**

    通配符说明¶

    通配符 说明 示例
    ** 匹配任意层级目录 04_研发内部设计/** 隐藏该文件夹下所有文件
    * 匹配单层目录 **/临时_*.md 隐藏所有以"临时_"开头的文档
    ? 匹配单个字符 docs/test?.md 隐藏 test1.md、test2.md 等

    常见隐藏场景¶

    exclude_docs: |
  # 隐藏单个文件夹的所有内容
  04_研发内部设计/**

  # 隐藏特定文件
  **/草稿.md

  # 隐藏某类文件（如所有临时文档）
  **/临时_*.md

  # 隐藏整个文件夹（包括子文件夹）
  内部资料/**

    注意事项¶

    1. 仅隐藏站点页面：exclude_docs 只会阻止文件生成到站点，文件本身仍在仓库中，源码提交时依然存在；
    2. 导航同步更新：隐藏后的文件夹不会再出现在站点导航栏中；
    3. 取消隐藏：删除或注释 exclude_docs 中对应的配置行即可恢复显示；
    4. 路径规则：路径相对于 docs_dir 配置的目录（默认为 docs/）。

    六、常见问题排查¶

    6.1 .gitignore 忽略 site/ 目录不生效¶

    问题现象

    已在 .gitignore 中添加了 site/，但 Git 客户端仍然显示 site/ 目录下的文件未暂存。

    根本原因：.gitignore 只对未被跟踪的文件生效。如果 site/ 在添加 .gitignore 之前已经被 git add 或 git commit 过，则忽略规则对该目录无效。

    解决方案：先从 Git 跟踪中移除（保留本地文件），再提交：

    # 从 git 跟踪中移除 site/ 目录（保留本地文件）
git rm -r --cached site/

# 提交移除
git add .
git commit -m "chore: 从 git 跟踪中移除 site/ 构建输出目录"
git push origin main

    执行后，site/ 文件将从远程仓库中删除，同时本地文件保留，之后 .gitignore 规则正常生效。

    预防建议

    项目初始搭建时，务必先创建 .gitignore 并确认包含 site/ 规则，再执行首次 git add . 提交，避免后续需要手动清理。

    6.2 MkDocs 构建时文档链接报 WARNING¶

    问题现象

    mkdocs build 或 mkdocs gh-deploy 时出现类似警告： 

    WARNING - Doc file 'xxx.md' contains a link '../path/file.md', 
          but the target 'path/file.md' is not found among documentation files.

    常见原因：文档中使用了相对路径链接（如 ../dir/file.md），但路径层级计算错误。

    排查方法：

    1. 确认链接起点的目录层级：如文件 04_研发内部设计/阳煤商品治理/xxx.md 深度为 2 层
    2. 确认目标文件的位置：如目标在 03_项目技术架构文档/file.md（深度 1 层）
    3. 计算正确的相对路径：
    起点目录 目标目录 正确相对路径
    04_研发内部设计/阳煤商品治理/ 03_项目技术架构文档/ ../../03_项目技术架构文档/file.md
    04_研发内部设计/ 03_项目技术架构文档/ ../03_项目技术架构文档/file.md
    docs/ 03_项目技术架构文档/ 03_项目技术架构文档/file.md

    核心原则

    • 每个 ../ 向上退一层目录
    • 从起点文件所在目录开始计算，直到两个文件的公共父目录

    8.2 文档头部标签化元数据¶

    每个文档 # 标题 之后必须紧跟标准化标签头部：

    # 文档标题

:material-file-document-edit: **文档类型**: 类型说明 |
:material-account-clock: **更新时间**: YYYY-MM-DD |
:material-account: **维护人**: 姓名 |
:material-tag: **标签**: 关键词1, 关键词2, 关键词3
    文档类型 专属字段
    故障排查类 :material-alert-circle: **优先级**: 🔴 高 + :material-account-clock: **发生时间**:
    功能设计类 :material-api: **涉及模块**: 模块名

    8.3 标签命名规范¶

    原则 说明 示例
    技术关键字优先 组件名、中间件名 Elasticsearch, HBase, Kafka
    项目/租户名 涉及的租户或客户 阳煤, 华能, 中海油
    功能/模块名 核心功能或服务模块 数据同步, 价格监测
    问题/场景标识 故障现象、操作类型 OOM, 写入拒绝, 任务卡死
    • 每个文档 3~6 个标签，按重要性从大到小排列
    • 标签之间用英文逗号 + 空格分隔

    8.7 Mermaid 图表支持¶

    在 mkdocs.yml 中配置了 Mermaid 支持，可使用以下语法绘制流程图：

    ```mermaid
flowchart TD
    A[开始] --> B[处理]
    B --> C[结束]
```

    8.8 安全红线：禁止输出真实密码¶

    安全红线：生成任何文档时，绝对禁止输出真实密码、密钥、Token、证书等敏感凭证信息。

    必须遵守的规则 示例
    密码/密钥占位 ❌ password=real123 → ✅ password=your_password
    连接字符串脱敏 ❌ jdbc:mysql://...?password=real123 → ✅ ...?password=your_password
    配置文件示例 ❌ spring.redis.password=MyRedis@2024 → ✅ spring.redis.password=your_redis_password

    违反后果

    文档一旦提交到 Git 仓库，密码将永久记录在版本历史中，即使后续删除也无法彻底清除。
