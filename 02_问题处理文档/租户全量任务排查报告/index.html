    租户全量任务卡死问题排查报告¶

    1. 问题描述¶

    1.1 现象¶

    • 任务状态: 租户全量任务卡在"站外商品价格数据同步中"（状态302）
    • 数据积压: 数据库中有约59万条数据未处理（crawl_state=0, callback_state=0）
    • MQ状态: callbackdata.queue 消息数极少（仅26条），且几乎不再变化

    1.2 初步排查¶

    -- 数据积压情况
SELECT crawl_state, callback_state, COUNT(*) 
FROM ent_acquisition_task_price_sync_data 
GROUP BY crawl_state, callback_state;
-- 结果: 587031 条数据 crawl_state=0, callback_state=0（待爬取）

    3. 详细排查过程¶

    3.1 第一阶段：确认数据积压位置¶

    -- 检查数据状态分布
SELECT crawl_state, callback_state, COUNT(*) as count 
FROM ent_acquisition_task_price_sync_data 
GROUP BY crawl_state, callback_state 
ORDER BY count DESC;

    结果: - crawl_state=0, callback_state=0: 587031条（待爬取） - crawl_state=0, callback_state=1: 极少（已爬取待回调） - crawl_state=2, callback_state=2: 完成的数据

    结论: 数据积压在爬取阶段，爬虫产出太慢。

    3.2 第二阶段：排查MQ队列¶

    # 查看callbackdata.queue队列状态
rabbitmqctl list_queues | grep callbackdata

    现象: callbackdata.queue 消息数仅26条，且长时间不变化。

    疑问: 数据库有59万条待爬取，为什么队列只有26条？

    原因: SourceMqService从crawldata队列消费消息后，调用爬虫同步阻塞爬取，爬完一条再发回调。爬虫卡住导致消费极慢。

    3.3 第三阶段：线程Dump分析¶

    # 获取goods-crawl服务进程ID
jps -lvm | grep goods-crawl
# 输出: 5637 goods-crawl.jar

# 获取线程Dump
jstack 5637 > goods-crawl-thread-dump.txt

    关键发现（在jstack输出中搜索BLOCKED/WAITING线程）:

    结论: 所有爬虫线程都在等待HTTP响应，没有空闲线程处理新任务。

    3.5 第五阶段：确认服务卡死¶

    连续执行以下命令观察队列:

    # 监控callbackdata.queue消息数
watch -n 5 'rabbitmqctl list_queues | grep callbackdata'

    现象: 消息数长时间保持在26条，几乎无变化。

    结论: 服务已完全卡死，无法继续消费MQ消息。

    :material-root: 4. 根因分析¶

    4.1 直接原因¶

    HTTP Socket读取阻塞: 目标网站响应极慢或挂起连接，导致爬虫线程永久等待在 SocketInputStream.read()。

    4.2 设计缺陷¶

    1. 单线程顺序爬取: AbstractPageSpider.innerCrawl() 使用 while 循环逐个处理请求，没有并行能力
    2. 无超时控制: HTTP请求未设置合理的连接超时和读取超时
    3. 线程资源耗尽: 16个爬虫线程全部被阻塞，没有空闲线程处理新任务
    4. 同步阻塞模式: 整个调用链是同步阻塞的，一条请求卡死会阻塞后续所有请求

    4.3 问题本质¶

    单线程设计 + 无超时控制 + 目标网站响应慢 = 雪崩效应
     │              │                │
     │              │                └── 外部不可控因素
     │              └── 内部可控，应设置5-10秒超时
     └── 应改为线程池并发爬取，提升吞吐量

    5. 解决方案¶

    5.1 临时解决方案（已执行）¶

    # 1. 停止卡死的goods-crawl服务
kill 5637

# 2. 重新启动goods-crawl服务
nohup java -jar goods-crawl.jar > goods-crawl.log 2>&1 &

# 3. 验证恢复
jps -lvm | grep goods-crawl
# 新PID: 15652

    5.2 长期优化建议¶

    5.2.1 设置HTTP超时¶

    // 在WebMagic配置中添加超时设置
Site site = Site.me()
    .setTimeOut(10000)  // 10秒超时
    .setRetryTimes(3)   // 重试3次
    .setCycleRetryTimes(3)
    .setSleepTime(1000);

    5.2.2 改为线程池并发爬取¶

    // 方案1：使用WebMagic的多线程能力
Spider.create(new PricePageProcessor())
    .thread(10)  // 10个线程并发
    .run();

// 方案2：将单线程innerCrawl改为线程池
public CrawlResult<T> innerCrawl(List<GoodsPageRequest> requests) {
    ExecutorService executor = Executors.newFixedThreadPool(10);
    List<Future<?>> futures = new ArrayList<>();

    for (GoodsPageRequest r : requests) {
        futures.add(executor.submit(() -> crawl(r, ...)));
    }

    // 等待所有任务完成
    for (Future<?> f : futures) {
        f.get(30, TimeUnit.SECONDS);  // 30秒超时
    }
}

    5.2.3 增加熔断机制¶

    // 使用Resilience4j或Hystrix添加熔断
@CircuitBreaker(name = "crawler", fallbackMethod = "fallback")
public CrawlSourceItem blockCrawl(...) {
    // 爬取逻辑
}

    :material-summary: 9. 总结¶

    问题总结¶

    1. 现象: 租户全量任务卡死，59万条数据待爬取
    2. 原因: 16个爬虫线程全部卡在HTTP Socket读取，无空闲线程处理新任务
    3. 根因: 单线程顺序爬取设计 + 无HTTP超时控制

    已采取措施¶

    1. 通过jstack诊断确认线程阻塞状态
    2. 重启goods-crawl服务（PID: 5637 → 15652）
    3. 验证恢复：callbackdata.queue消息数开始波动

    后续建议¶

    1. 短期: 监控队列状态，确保任务正常推进
    2. 中期: 添加HTTP超时配置，避免永久阻塞
    3. 长期: 重构爬虫为线程池并发模式，提升吞吐量
