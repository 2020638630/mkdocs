# :material-message-alert: 价格监测平台 - 消息生成链路说明文档

---

## :material-information: 一、问题概述#

**现象**：执行价格监测任务后，`t_developer_msg_detail` 表没有消息数据。

**根因**：`goods_source.generate_msg = 0` 配置导致消息未生成。

---

## :material-cog: 二、关键配置说明#

| 配置项 | 位置 | 说明 |
|-------|------|------|
| `generate_msg` | `goods_source` 表 | 消息生成总开关：<br>• `0` = 不生成消息<br>• `1` = 全部生成<br>• `2` = 仅API商品生成 |
| `tenant_type` | `sys_tenant` 表 | 租户类型：`private`=私有化，其他=SaaS |
| `useNewFlow` | 任务配置 | 是否走新流程（Kafka回调） |

---

## :material-sitemap: 三、消息生成完整链路

### 阶段1：任务构建（XXL-Job定时器）#

**入口**：`XxlJobController.buildPriceMonitorTask()`

```
XXL-Job定时器
    ↓
buildPriceMonitorTask() 构建监测任务
    ↓
生成任务ID，调用大数据服务
```

---

### 阶段2：大数据计算#

**入口**：大数据服务 `TaskStrategy.run()`

```
TaskStrategy.run()
    ↓
PriceStatService.priceStat() 价格统计计算
    ↓
计算合理价、预警价等
    ↓
HTTP回调：通知运营端任务完成
```

**说明**：此阶段计算商品价格数据，不直接涉及消息生成。

---

### 阶段3：数据同步（运营端 → 租户端）#

**入口**：`TaskController.taskOver()`

```
TaskController.taskOver() 任务完成回调
    ↓
ForEntpurSyncServiceImpl.syncGoodsForEntpur()
    ↓
遍历商品数据，构建同步对象
    ↓
buildGoodsPriceStatItem() - 【关键逻辑】
    ↓
RabbitMQ发送 queue.monitor.result
```

**关键代码**（`ForEntpurSyncServiceImpl.java:532-543`）：

```java
Integer generateMsg = tenantSource.getGenerateMsg();
if (generateMsg == 0) { 
    // 配置为0，不生成消息
    goodsWithPriceStatItem.setGenerateMsg(false);
} else if (generateMsg == 1) { 
    // 配置为1，生成消息
    goodsWithPriceStatItem.setGenerateMsg(true);
}
```

---

### 阶段4：租户端消费消息#

**入口**：`MonitorResultConsumer.MonitorResultConsumerListener()`

```
MonitorResultConsumerListener 监听MQ消息
    ↓
goodsMergeInfoService.saveGoodsMergeInfo()
    ↓
【关键判断】是否生成消息
    ↓
条件满足 → generateMsgDetail() 生成消息
```

**关键判断逻辑**（`GoodsMergeInfoServiceImpl.java`）：

```java
// 只有同时满足以下条件才生成消息：
// 1. 不是私有化租户（!"private".equals(tenantType))
// 2. isGenerateMsg = true（由阶段3的generate_msg配置决定）
if (!"private".equals(sysTenantDTO.getTenantType())) && isGenerateMsg) {
    generateMsgDetail();
}
```

---

### 阶段5：保存消息到数据库#

**入口**：`TDevMsgDetailServiceImpl.transTDMsgDetails()`

```
generateMsgDetail() 构建消息内容
    ↓
tDevMsgDetailService.transTDMsgDetails()
    ↓
INSERT INTO t_developer_msg_detail
```

## :material-message-question: 四、两种"消息"的区别#

| 类型 | 载体 | generate_msg控制 | 说明 |
|------|------|-----------------|------|
| **MQ消息** | RabbitMQ `queue.monitor.result` | :x: 始终发送 | 任务结果通知，与配置无关 |
| **业务消息** | `t_developer_msg_detail` 表 | :white_check_mark: 受配置控制 | 开发者可见的预警/调价消息 |

---

## :material-calculator: 五、合理价为0的处理逻辑#

**结论**：合理价为0 **不会阻止** 消息生成。

**处理逻辑**：
1. 合理价为0时，会添加错误码 `GOODSERROR0002`（合理价计算异常）
2. 商品数据仍会被保存到 `t_goods_merge_his` 表
3. 消息生成只受 `generate_msg` 配置控制

**示例数据**：
```sql
reasonable_price = 0
goods_errors = 'GOODSERROR0002,GOODSERROR0003,GOODSERROR0005'
generate_msg = 0  ← 这才是消息未生成的原因
```

## :material-wrench: 六、解决方案#

### 立即修复#

```sql
-- 开启消息生成（全部生成）
UPDATE goods_source 
SET generate_msg = 1 
WHERE tenant_id = '你的租户ID';

-- 验证配置
SELECT tenant_id, generate_msg 
FROM goods_source 
WHERE tenant_id = '你的租户ID';
```

---

### 重新执行监测任务#

1. 进入运营端管理后台
2. 找到对应监测任务
3. 点击"重新执行"或等待定时器触发
4. 检查 `t_developer_msg_detail` 表是否有新数据

## :material-clipboard-text: 九、总结

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| RabbitMQ宕机 | 系统内存耗尽，Erlang VM崩溃 | 添加Swap + 重启服务 |
| 系统内存耗尽 | Redis maxmemory=25GB过高 | 降低至12GB |
| 进程被杀死 | 无Swap空间，OOM Killer触发 | 添加8GB Swap |
| 配置不持久 | Docker容器权限不足 | 修改宿主机挂载配置文件 |
| DB2内存占用11.91G | GOODSCRAWL缓存TTL 3-4天，2200万键 | 缩短TTL至1天（待改代码） |
| CATALOGMAP永不过期 | 代码未设TTL | 添加expire参数（待改代码） |
